# Task: agmon stage 3 — the CLI

agmon's API is complete but uncurated; querying it means hand-writing curl
and jq against JSON whose shape you have to discover by querying it. Stage
3 adds the editorial layer: an `agmon` command-line client with curated
default views, layered disclosure, and full-fidelity escape hatches. It
also completes the single-package story: the server and the launch wrapper
become subcommands of the same installable tool.

Read CLAUDE.md, the README, and the existing code first. BACKLOG.md items
are deferred by policy — do not fix them opportunistically.

## Package changes

- Add a console script: `agmon = "agmon.cli:main"`. `python -m agmon`
  becomes equivalent to `agmon` (the CLI), and serving moves to
  `agmon serve`. Update the README and add `deploy/agmon.service` (a
  user-unit example using `agmon serve`) so operators can update their
  unit; note the required unit change prominently in the README.
- New runtime deps: `httpx`, `rich` (plus existing fastapi/uvicorn).
- New modules: `agmon/client.py` (pure API client: takes a base URL,
  returns parsed dicts, no printing), `agmon/render.py` (all formatting),
  `agmon/cli.py` (argument parsing and wiring). Keep client and render
  importable and terminal-independent — a future TUI and an Emacs bridge
  will reuse them. cli.py must accept an injected client and output
  writer so tests run with no network and no TTY.

## Global behavior

- Server address: `--url` flag, else `$AGMON_URL`, else
  `http://localhost:8400`.
- Run-id arguments are optional almost everywhere: omitted means the most
  recently started run. IDs match by unique **substring**, not prefix —
  run ids are `<timestamp>-<hex>`, so the distinguishing part is the hex
  suffix and the date prefix is shared by every run that day; `agmon show
  a3f9` must work. An exact full-id match always wins outright; otherwise
  a substring matching exactly one run resolves to it; ambiguous
  substrings error listing the candidates; no match errors clearly.
- Output layers: when stdout is a TTY, rich tables/formatting; when
  piped, plain TSV with no decoration (`--plain` forces this on a TTY).
  `--json` on every read command emits the full underlying object(s).
- Field discovery: read commands accept `--fields a,b,c` to project
  specific fields into the table/TSV/JSON output. Field names address the
  JSON flattened one level with dots (e.g. `status.effective_status`,
  `metrics.total_cost_usd`). Bare `--fields` (no value) prints the
  available field names for that command and exits — the schema is
  discoverable from the tool itself.
- Times render as relative ("3m ago") in tables; durations as "12m40s".
  Raw ISO stays available via `--json`/`--fields`.
- `agmon --version` works.

## Commands

**`agmon ls`** — fleet glance. Newest first, default 20 rows
(`-n`, `--all`). Columns: short id, effective_status (color-coded:
running cyan, finished green, error/died red, interrupted/stalled
yellow), started (relative), duration, project (basename of cwd), last
activity (tool + truncated target), issues, cost. Filters: `--status`,
`--session <sid>`.

**`agmon show [id]`** — one run, digested. Sections, in order: header
(id, session id, status, started/ended/duration, turns, cost, model, cwd,
git branch); lineage — computed client-side by grouping the run list on
session_id — as "resumed from <id>" / "resumed by <id>" lines when
applicable; prompt (first ~6 lines, `--full-prompt` for all); latest
PROGRESS line; last tool; issues (count, then up to 5 entries); and the
run's final result text, rendered as markdown (`--raw` to disable
rendering). Uses the summary endpoint.

**`agmon tail [id]`** — live follow. Polls
`/v1/runs/{id}/events?after=<cursor>` every 2s from seq 0 (so tailing a
finished run replays it and exits — this is intended). Rendering, shared
with `events` via render.py: tool_use as dim tool name + target truncated
to terminal width; PROGRESS lines highlighted; error events red with a
short snippet; assistant text as a single dim truncated line; system/init
as one line. On a result event or terminal status: print a summary line
(status, cost, duration) and exit with code 0 for finished, 1 for error,
3 for died — making `agmon tail $id && next-thing` scriptable. If the
run is running but eventless past the stall threshold, print a yellow
"stalled for Xs" heartbeat (keep polling; stalled is not terminal).
`--last N` starts the cursor N events from the end instead of 0.

**`agmon events [id]`** — forensics. Table: seq, time, type/subtype,
compact payload summary (same renderer). Flags: `--errors-only`,
`--type`, `--after <seq>`, `-n`. `--json` emits raw payloads.

**`agmon costs`** — wraps `/v1/stats/costs`. Bucket table plus a totals
row; `--days N` (default 30) or explicit `--since`/`--until`.

**`agmon serve`** — starts the collector + API exactly as `python -m
agmon` does today; `--host`/`--port` override the env vars.

**`agmon run`** — verbatim port of the `agmon-run` wrapper script
currently installed at `~/.local/bin/agmon-run` (read it; port it, do not
redesign it). Same flags, same spool behavior, same PROGRESS
system-prompt injection, same git author env, still prints the run_id.
Behavior changes are out of scope; add only arg-parsing smoke tests.
Document in the README that `run` and `serve` execute box-side, while
read commands work from anywhere with `$AGMON_URL` set.

## Server-side changes (only these)

1. The summary response gains `result_text`: the full `result` string
   from the run's result event, null if absent. (BACKLOG has no entry for
   this; it was an agreed roadmap item.)
2. `/healthz` moves back to unversioned (health is an operational
   endpoint with a different stability contract than the data API).
   Remove `/v1/healthz`. Update README and any examples.

## Tests (required, no network, no TTY dependence)

1. render: compaction of realistic tool_use / PROGRESS / error / result
   payloads (use shapes lifted from real spool events).
2. client: id resolution — unique substring hit (a hex-suffix fragment),
   an ambiguous fragment shared by multiple ids (e.g. a date prefix)
   erroring with candidates listed, exact full-id match winning, and
   no-match erroring; omitted id resolves to latest.
3. lineage: resumed-from/by computed correctly from a fixture run list
   sharing session_ids.
4. tail loop: with a stubbed client yielding event batches, the cursor
   advances via next_after, rendering is called per event, and terminal
   statuses map to the documented exit codes (all of: finished, error,
   died) without real sleeping — make the poll delay injectable.
5. fields: one-level dot flattening; bare `--fields` lists names.
6. TSV vs table selection via the injected TTY flag; `--plain` forces
   TSV.
7. server: summary includes `result_text`; `/healthz` responds
   unversioned and `/v1/healthz` is 404.
