# Task: agmon — run deletion (`agmon rm` + `DELETE /v1/runs/{id}`)

The spool grows forever: probe runs, auth-failure duds, and abandoned
experiments accumulate beside the runs worth keeping. Add permanent
per-run deletion by id, usable from anywhere a client can reach the
server — not only box-side.

Deletion is the first operation that destroys part of the source of
truth, so the architecture's standing invariants bind hard: the spool is
the source of truth, the SQLite DB is a disposable index, and the
ingester thread is the **sole DB writer**. Deletion therefore removes
**spool files only** — the HTTP handler never touches the DB (handlers
stay read-only with respect to it), and the index heals itself via
ingester reconciliation. The delete route is the server's first and only
spool mutation; keep it that narrow. Repo conventions in CLAUDE.md
apply, including test-first feature work.

## 1. HTTP: `DELETE /v1/runs/{run_id}`

- Takes the **exact full run id** (resolution is a client concern, as
  everywhere else). Unknown id — nothing in the index and no spool files
  — is 404 with `{"error": ...}`.
- Refuses a live run: meta `status: "running"` **and** pid alive → 409
  with an error saying to kill it first (no force; run-control is out of
  scope). A died run (meta `running`, pid gone) is deletable — that is
  precisely the junk this command exists for. Deleting files the wrapper
  still holds open would not stop it and would silently discard its
  output; the pid check is the guard.
- On success: remove `<id>.meta.json`, `<id>.jsonl`, `<id>.stderr.log`
  (any already missing is fine) and return `{"deleted": run_id}`.
  Permanent — there is no trash.
- Deletion is **idempotent** — per DELETE semantics and by construction:
  the route only makes files disappear, `unlink` of an already-vanished
  name is "already done" (POSIX unlink is atomic; concurrent deletes
  race harmlessly), and a meta that disappears mid-request — a
  concurrent delete won — means "nothing left to refuse, keep going."
  Every interleaving converges without a lock. A repeat DELETE inside
  the reconciliation lag returns 200 again; after the purge it is 404;
  a client that resolved an id and then gets 404 on the DELETE reports
  "already gone" as success, not an error.
- The run leaves the index on the **next scan** (§3), so reads may still
  return it for ~one scan interval after a 200. Document the lag; clients
  must not treat brief presence-after-delete as failure.
- Trust note: this is the API's first destructive route, on a server that
  has no auth by design (tailnet trust model). That stays the model; the
  spec adds no auth.

## 2. CLI: `agmon rm <id>`

A remote command like `show`/`tail` — works anywhere `$AGMON_URL`
reaches; needs the server up (deletion has one path, through the route).

- The id is **required** — no omitted-means-latest default; a destructive
  command never guesses. Resolution by unique substring via the existing
  pure resolver against the server's run list (exact full id wins;
  ambiguous errors listing candidates; unknown errors).
- Before deleting, print a one-line summary of the target (id, status,
  started_at) and require confirmation on a TTY. `--yes`/`-y` skips the
  prompt; a non-TTY invocation **without** `--yes` errors and deletes
  nothing — no silent destruction in scripts.
- On success, report the deletion and the ~one-scan lag before the run
  leaves list views.

## 3. Ingester: reconcile vanished runs

On each scan, purge the index rows of any tracked run whose meta **and**
jsonl are both gone from the spool dir: `runs`, `events`, `run_labels`,
and `ingest_state`, in one transaction. This is the sole-writer doing the
deletion, so the single-writer invariant holds — regardless of whether
the files vanished via the route or an operator's `rm` in the spool dir.

- A run with only one file missing is **not** purged (a meta-only run —
  e.g. killed pre-spool — is a legitimate live state, and a lone missing
  file is not the deletion signature).
- A deletion racing an in-flight scan is harmless: rows written for a
  just-deleted run are healed by the next scan. Idempotent by design.
- A crash-partial delete (handler died between unlinks: e.g. meta gone,
  jsonl present) is deliberately **not** purged — the single-file rule
  above protects legitimate states, and the repair tool is the route's
  own idempotency: re-issue the DELETE and the remaining files go.
- Reconciliation runs against the live directory, so **no schema bump**:
  no derivation over surviving rows changes, and stale rows from
  deletions performed under old code are healed by the first new-code
  scan. If you believe you need a bump, stop and ask.

## Tests (test-first, per conventions)

1. Route: DELETE on a finished run removes all three spool files (a
   missing `stderr.log` doesn't error) and returns `{"deleted": id}`.
2. Route: unknown id → 404; meta `running` + pid alive → 409 and files
   untouched; meta `running` + dead pid (died) → deletes.
3. Route + reconciliation: after a 200, the next `scan()` purges the
   run's `runs`, `events`, `run_labels`, and `ingest_state` rows.
4. Reconciliation is surgical: with two ingested runs, deleting one's
   files leaves the other's rows (and byte offsets) intact.
5. Partial vanish: jsonl deleted but meta present → rows stay; the run
   still lists.
6. CLI resolution: unique substring resolves; ambiguous errors listing
   candidates; exact full id wins over a substring-superset; unknown id
   errors. Id required: bare `agmon rm` is a usage error.
7. CLI confirmation: non-TTY without `--yes` errors and calls no delete;
   `--yes` deletes (stub client records the call).
8. Idempotency: a second DELETE for the same id succeeds while the run
   is still indexed (lag window) and 404s after the purge; a
   half-deleted run (meta removed, jsonl left behind) is finished off by
   a repeat DELETE. The CLI treats a 404 on DELETE after successful
   resolution as "already gone" (exit 0).

## Definition of done

- Full suite green; BACKLOG xfails undisturbed. No new columns, no
  schema bump; the delete route is the only new HTTP surface and the
  only code path that removes spool files.
- Deleting a run with the server up: it leaves `agmon ls` within one
  scan interval.
- README and CLAUDE.md updated per §4. All work committed, tree clean.

Out of scope: bulk deletion (`--older-than`, `--status`, globs), any
trash/archive/undo mechanism, killing running runs (stage-4 run-control),
auth on the API, and agmon.el integration (a later spec rides on this
route).
